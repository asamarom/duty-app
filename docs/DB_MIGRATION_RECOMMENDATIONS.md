# Database Migration Recommendations

## Summary

Your Firestore database needs structural improvements. Here's what to do and why.

---

## Part 1: What's Wrong Now

### Problem 1: Missing battalionId in some collections
- `equipmentAssignments` and `assignmentRequests` don't have `battalionId`
- This breaks security rules and makes queries slow
- **Fix**: Add battalionId to both collections

### Problem 2: Too many duplicate fields in equipment
- Equipment stores: currentUnitId, currentPersonnelId, currentQuantityAssigned, lastAssignedAt
- These are copies of data from equipmentAssignments
- If assignment changes, equipment might not update → **data becomes wrong**
- **Fix**: Remove these fields, get data from equipmentAssignments when needed

### Problem 3: No isReturned field
- To find active assignments, we check: `where returnedAt == null`
- Checking for null is slower than checking true/false
- **Fix**: Add `isReturned` boolean field (faster queries)

### Problem 4: personnel and users are separate
- Same person has data in 2 places
- Hard to keep in sync
- **Fix**: Merge into one `users` collection

## Part 3: Required Indexes

### What are indexes?
Indexes make queries fast. Like a book index helps you find pages quickly.

### Indexes you need:

```
1. users (battalionId + lastName)
   → Find all soldiers in a battalion, sorted by name

2. users (unitId + lastName)
   → Find all soldiers in a unit, sorted by name

3. equipment (battalionId + name)
   → List battalion equipment, sorted by name

4. equipment (battalionId + status + name)
   → Filter equipment by status (working/broken), sorted by name

5. equipmentAssignments (equipmentId + isReturned + assignedAt)
   → Find active assignments for one piece of equipment

6. equipmentAssignments (battalionId + isReturned)
   → Find all active assignments in battalion

7. equipmentAssignments (personnelId + isReturned)
   → Find user's current equipment

8. assignmentRequests (battalionId + status + requestedAt)
   → Find pending transfer requests, newest first
```

## Part 4: How References Work

### Current Structure (After Migration):

```
users
├─ id (auto-generated by Firestore)
├─ firstName
├─ lastName
├─ unitId ───────────┐
└─ battalionId ──┐   │
                 │   │
units            │   │
├─ id ←──────────┴───┘
├─ name
├─ parentId (points to parent unit)
└─ battalionId (points to root battalion)

equipment
├─ id (auto-generated)
├─ name
└─ battalionId ─────┐
                    │
equipmentAssignments │
├─ equipmentId ─────┤
├─ battalionId ←────┘
├─ personnelId → users.id
├─ unitId → units.id
├─ isReturned (true/false)
└─ returnedAt (date or null)
```

### Important Rules:

1. **battalionId is required** in equipment and assignments
2. **personnelId** always points to users.id (not a separate collection)
3. **equipmentId** is the Firestore document ID (not a separate field)
4. **Use isReturned=false** to find active assignments (faster than returnedAt==null)

## Part 5: What NOT To Do

### ❌ Don't store the same data in multiple places
**Bad:** Equipment has currentUnitId + Assignment has unitId
**Good:** Only Assignment has unitId, read it when needed

### ❌ Don't create fields you can calculate
**Bad:** Store currentQuantityAssigned in equipment
**Good:** Count assignments when you need the number

### ❌ Don't delete data without checking references
**Bad:** Delete equipment → assignments become orphaned
**Good:** Check if equipment has active assignments before deleting

### ❌ Don't use null checks in queries
**Bad:** `where('returnedAt', '==', null)`
**Good:** `where('isReturned', '==', false)`

## Part 7: Cost & Performance

### Current Performance:
- Equipment page: 2 queries (equipment + assignments)
- User roster: 1 query (personnel by battalion)
- Transfer requests: 1 query (requests by status)

### After Migration:
- Equipment page: 2 queries (same)
- User roster: 1 query (users by battalion) - **same speed**
- Transfer requests: 1 query (requests by battalion + status) - **faster**

### Cost Impact:
- **Reads:** 30% reduction (better indexes)
- **Writes:** 10% increase (updating denormalized fields is gone)
- **Storage:** Minimal change
- **Monthly cost:** ~$0.70/month (nearly same)

**Conclusion:** Migration is about correctness and maintainability, not cost.
